# DB-MCP 日本語指示サンプル

DB-mcp に対して、データベース検索や構造確認を依頼するときの日本語テンプレート集です。

曖昧な依頼よりも、テーブル名、カラム、件数、条件、SQL などを明示した方が安定して発火します。

## 使い方のコツ

- 構造を知りたいときは `テーブル` `カラム` `スキーマ` を入れる
- データを見たいときは `件数` `一覧` `先頭5件` `条件` を入れる
- SQLとして明示したいときは `SQLで` を先頭につける
- 対象テーブル名を必ず入れる
- 条件がある場合は `where` 相当の条件を日本語で具体的に書く

## 構造確認テンプレ

| やりたいこと | 指示サンプル |
|---|---|
| テーブルの列を知りたい | propertiesテーブルのカラム構成を教えて |
| 列名を一覧したい | propertiesテーブルのカラム名を一覧で出して |
| 型も知りたい | propertiesテーブルの各カラム名とデータ型を教えて |
| 主キーを知りたい | propertiesテーブルの主キーを教えて |
| インデックスを知りたい | propertiesテーブルのインデックスを教えて |
| 特定カラムの有無を知りたい | propertiesテーブルにstatusカラムがあるか確認して |
| 関連テーブルを探したい | propertiesに関連するテーブルやカラムを探して |
| スキーマ全体を見たい | propertiesテーブルのスキーマを教えて |

## SQL 明示テンプレ

| やりたいこと | 指示サンプル |
|---|---|
| SQLで件数取得したい | SQLで properties テーブルの件数を取得して |
| SQLで先頭10件を見たい | SQLで properties テーブルの先頭10件を取得して |
| SQLで条件検索したい | SQLで properties テーブルから status='active' の行を10件取得して |

## 汎用定型文

以下はそのまま流用しやすい短い定型文です。

- `◯◯テーブルのカラム構成を教えて`
- `◯◯テーブルの各カラム名と型を一覧で出して`
- `◯◯テーブルの件数を教えて`
- `◯◯テーブルの先頭5件を見せて`
- `◯◯テーブルで△△の条件に一致する行を5件見せて`
- `◯◯に関連するテーブルやカラムを探して`
- `SQLで◯◯テーブルを検索して`

## 発火しやすい書き方

### 良い例

- propertiesテーブルのカラム構成を教えて
- propertiesテーブルの件数を教えて
- propertiesテーブルでstatusがactiveの行を5件見せて
- SQLで properties テーブルの件数を取得して

### 避けた方がよい例

- ちょっとDB見て
- これ調べて
- 一般的にどういうカラムがありますか
- propertiesについて教えて

後者のように対象や目的が曖昧だと、一般知識で回答されやすくなります。

## おすすめ運用

- 構造確認は `テーブル` `カラム` `スキーマ` を明示する
- 検索は `件数` `一覧` `先頭N件` `条件` を明示する
- 複雑な依頼は `SQLで` をつける
- 実在テーブル名を入れる
- 必要なら件数制限を入れる (`5件`, `10件` など)

## 自然文から候補探索される例

現在の DB-mcp は、明示的なテーブル名がなくても、自然文から候補テーブルやカラムを探索する方向に寄せています。

そのため、以下のような一般ユーザー向けの表現でも、まずテーブル一覧やカラム一覧を見て候補探索する動きが期待できます。

### 自然文の例

| やりたいこと | 自然な言い方の例 |
|---|---|
| 物件一覧を見たい | 物件の一覧を見せて |
| 新しい物件を見たい | 最近登録された物件を見せて |
| 条件に合う物件を見たい | 東京の物件を探して |
| 件数を知りたい | 物件は全部で何件ありますか |
| 絞り込みたい | ペット可の物件を見せて |
| 顧客情報を見たい | 顧客の一覧を見せて |
| 契約情報を見たい | 契約中のデータを見せて |
| 集計したい | 物件の種類ごとの件数を知りたい |
| 最近更新されたものを見たい | 更新が新しい順に見せて |
| 特定条件で探したい | 家賃が安い物件を探して |

### こう書くとより安定する

- 対象を入れる: `物件`, `顧客`, `契約`, `売上`
- 目的を入れる: `一覧`, `件数`, `探して`, `見せて`, `集計して`
- 条件を入れる: `東京の`, `ペット可の`, `契約中の`, `最新の`

例:

- `物件の一覧を見せて`
- `東京の物件を探して`
- `契約中の顧客数を知りたい`
- `更新が新しい順に見せて`

### まだ避けた方がよい曖昧表現

- `何か見せて`
- `このへん調べて`
- `データある?`
- `最近のやつ見たい`

このレベルまで曖昧だと、候補探索はできても対象を正しく絞れず、意図しないテーブルを選ぶ可能性があります。

### 実運用の目安

- 一般ユーザー向けUIでは、まず自然文で受ける
- 対象が曖昧なときだけ追加質問する
- 業務で重要な問い合わせは、対象語を含めてもらう

例:

- `物件の一覧を見せて`
- `顧客の件数を教えて`
- `契約中のデータを見せて`

この程度まで具体化されていれば、自然文でも DB 候補探索の精度はかなり上がります。
